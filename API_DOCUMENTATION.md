## Overview

This distributed medical system consists of two microservices:

- **Human_Project (Port 5000):** Core business logic for users, appointments, and patient records
- **Secure_System (Port 6000):** Encrypted medical file storage and management

Most endpoints require JWT authentication via Bearer token. The user registration endpoint (`POST /api/users`) is publicly accessible for user creation.

## Authentication & Authorization

### JWT Token Structure

```json
{
  "sub": "user_id",
  "email": "user@example.com",
  "role": "admin|doctor|patient",
  "iat": 1234567890,
  "exp": 1234571490
}
```

### Request Header

```
Authorization: Bearer <JWT_TOKEN>
```

### Token Acquisition

- Generated by the local `POST /auth/token` endpoint
- No refresh endpoint is implemented in the current API

### Token Generation Example

```
POST /auth/token
Content-Type: application/json
```

```json
{
  "sub": "1",
  "email": "user@example.com",
  "role": "doctor"
}
```

**Response:**

```json
{
  "message": "Token generated successfully",
  "token": "Bearer <JWT_TOKEN>"
}
```

### Failed Authentication Response

```json
{
  "status": "error",
  "msg": "Unauthorized - Invalid or missing token",
  "data": null
}
```

## Role Definitions

| Role | Level | Permissions | Use Case |
|------|-------|-------------|----------|
| **Admin** | 🔴 Highest | Full system access, user management, all CRUD operations | System administrator |
| **Doctor** | 🟡 Medium | Create/manage patient records, upload files, view appointments | Healthcare provider |
| **Patient** | 🟢 Lowest | Book appointments, access own records/files | Patients receiving care |

### Role Hierarchy & Access Rules

```
ADMIN (Unrestricted)
├── Can access all users, appointments, records, files
├── Can create records for any patient
├── Can delete any record
└── Cannot delete self or other admins

DOCTOR (Patient-scoped)
├── Can create records for assigned patients only
├── Can upload files for their patients
├── Can view appointments they're involved in
└── Cannot access other doctors' patient data

PATIENT (Self-scoped)
├── Can book appointments with available doctors
├── Can view own records
├── Can download own encrypted files
└── Cannot access other patients' data
```

### 🔐 MEDICAL FILES RESOURCE (`/api/files`)

#### Security Specifications

**Allowed File Types:**
- `.pdf` - Medical documents
- `.jpg`, `.jpeg`, `.png` - Medical images
- `.docx` - Word documents

**Blocked Extensions:**
- `.exe`, `.php`, `.js`, `.bat`, `.sh`, `.py`, `.rb`, `.ts` (executables prohibited)

**File Size Limits:**
- Maximum: **10 MB** per file

**Encryption:**
- Algorithm: **AES-256-CBC**
- Key: `FILE_ENCRYPTION_KEY` environment variable
- All files encrypted on disk

**Integrity Verification:**
- Hash Algorithm: **SHA-256**
- Verification on every download

**Storage:**
- Location: `private_uploads/` directory
- Persistence: Docker volume (`private_uploads`)

#### 4️⃣ GET Download Medical File

```
GET /api/files/:id/download
```

**Authentication:** ✅ Required  
**Role Required:** `admin` OR file owner  
**URL Parameters:** `id` (file ID)  
**Description:** Download decrypted file with integrity verification

**Process:**
1. Verify user has access to file
2. Decrypt file from storage
3. Calculate SHA-256 hash of decrypted content
4. Compare with stored hash (verify integrity)
5. Return file with original filename

**Response Headers:**
```
Content-Type: application/pdf
Content-Disposition: attachment; filename="report.pdf"
Content-Length: 512000
X-File-Hash: a1b2c3d4e5f6...
```

**Response Body:** Binary file content (decrypted)

**Error: Integrity Check Failed**
```json
{
  "status": "error",
  "msg": "File integrity check failed - possible corruption",
  "data": null
}
```

## Error Handling

### Standard Error Format

All error responses follow this structure:

```json
{
  "status": "error",
  "msg": "Human-readable error message",
  "data": null
}
```

### HTTP Status Codes

| Code | Meaning | Common Causes |
|------|---------|---------------|
| **200** | OK | Successful GET/PUT requests |
| **201** | Created | Successful POST requests |
| **400** | Bad Request | Invalid parameters, missing required fields |
| **401** | Unauthorized | Missing token, invalid token, expired token |
| **403** | Forbidden | Valid token but insufficient permissions |
| **404** | Not Found | Resource doesn't exist (user, appointment, etc.) |
| **422** | Unprocessable Entity | Validation error (invalid data format) |
| **500** | Server Error | Database error, server crash |

### Common Error Scenarios

**Missing Authentication Token:**
```json
{
  "status": "error",
  "msg": "Unauthorized - Token is required",
  "data": null
}
```
Status: **401**

**Invalid Role Access:**
```json
{
  "status": "error",
  "msg": "Forbidden - You don't have permission to access this resource",
  "data": null
}
```
Status: **403**

**Validation Error:**
```json
{
  "status": "error",
  "msg": "Validation failed: Email must be valid format",
  "data": null
}
```
Status: **400**

**Resource Not Found:**
```json
{
  "status": "error",
  "msg": "Appointment with ID 999 not found",
  "data": null
}
```
Status: **404**

## Security Measures

### 1. Authentication & Authorization

✅ **JWT-based Authentication**
- Tokens generated by the local `/auth/token` endpoint for development/testing
- Bearer token required in Authorization header
- Token expiration enforced

✅ **Role-Based Access Control (RBAC)**
- Three roles: admin, doctor, patient
- Endpoint-level role checking
- Resource-level ownership verification

### 2. Data Encryption

✅ **File Encryption (Secure_System)**
- AES-256-CBC encryption algorithm
- Files encrypted before disk storage
- Encryption key from environment variable

✅ **Password Security**
- Passwords hashed with bcrypt
- Never stored in plaintext
- Updated passwords re-hashed

### 3. Data Integrity

✅ **Hash Verification**
- SHA-256 hashing for uploaded files
- Hash verified on download
- Detects file corruption or tampering

### 4. Access Control

✅ **Row-Level Security**
- Users can only access their own data (except admin)
- Doctors can only manage own patients
- Patients can only view own records

✅ **File Isolation**
- Encrypted files in isolated directory
- Access controlled via database permissions
- Physical separation from application code

### 5. Input Validation

✅ **Request Validation**
- Email format validation
- Required field checking
- File type whitelisting
- File size limits

✅ **Database Constraints**
- Foreign key enforcement
- Unique constraints on emails
- Data type validation

### 6. Audit & Logging

✅ **Activity Logging**
- File uploads logged
- Access attempts recorded
- Errors logged with context

### 7. Environment Security

✅ **Sensitive Data**
- JWT_SECRET stored in environment

## Common Use Cases

### 1. Patient Books Appointment

**Flow:**
```
1. Patient calls GET /api/users/doctors (role=patient) - Get available doctors
2. Patient calls POST /api/appointment - Create appointment
   - doctor_id: Selected doctor
   - appointment_date: Chosen time
   - reason: Chief complaint
3. System returns appointment ID
```

### 2. Doctor Creates Patient Record

**Flow:**
```
1. Doctor calls GET /patient_Record (role=doctor) - View own records
2. Doctor calls POST /patient_Record - Create new record
   - patient_id: Patient ID
   - diagnosis: Medical diagnosis
   - prescription: Treatment
   - is_draft: true (for draft status)
3. Doctor calls PUT /patient_Record/:id - Finalize record
   - is_draft: false
```

### 3. Doctor Uploads Medical File

**Flow:**
```
1. Doctor calls POST /api/files/upload - Upload encrypted file
   - file: PDF/Image binary data
   - patient_id: Patient
   - file_type: xray/report/etc
   - appointment_id: Related appointment (optional)
2. File encrypted and stored in private_uploads/
3. Metadata stored in database
4. Patient can later download via GET /api/files/:id/download
```

### 4. Patient Accesses Own Medical Files

**Flow:**
```
1. Patient calls GET /api/files (role=patient) - See own files
2. Patient calls GET /api/files/:id/download - Download decrypted file
   - File decrypted from storage
   - Integrity verified (SHA-256)
   - Returned as attachment
```

### 5. Admin Manages Users

**Flow:**
```
1. Admin calls GET /api/users - View all users
2. Admin calls DELETE /api/users/:id - Remove user
   - Cannot delete self or other admins
3. Admin calls PUT /api/users/:id - Update user
   - Can change roles for other users
```

## Support & Troubleshooting

### Common Issues

**401 Unauthorized:**
- Check JWT token in Authorization header
- Verify token hasn't expired
- Check token format (Bearer <token>)

**403 Forbidden:**
- Verify user role has permission
- Check resource ownership
- Confirm user ID matches resource owner

**400 Bad Request:**
- Validate request body JSON syntax
- Check required fields are present
- Verify data types match specification

**File Upload Issues:**
- Check file size (max 10MB)
- Verify file extension is allowed
- Ensure patient_id exists
